Knowledge-base maintenance redesign

Replacing the crunch god-job with a model organised around two axes the original design ignored: what triggers the work, and how much data the work needs to look at.

Proposal 2026-06-23 Companion: maintenance-redesign.md

01The problem

The pain — "a confusing stream of PRs" — comes almost entirely from one job. Crunch has no scope (it ingests the whole KB) and no single responsibility, so it competes with the gap job, competes with itself, and emits PRs nobody can reason about.

Job Trigger Scope of data Verdict
source_change_sync source commit diff + ≤6 matching docs well-scoped
gaps-to-pull-requests every 10 min gap clusters mostly fine
refresh_pull_requests every 5 min open PRs plumbing
trigger_scheduled_crunch daily 02:00 all docs the god-job

The root cause isn't "too many jobs". It's that the work was never decomposed by trigger and scope. Crunch is five jobs wearing one coat.

02The reframe: trigger × scope

Every maintenance action is either event-driven (the world changed) or patrol (the KB rots on its own). And nothing needs the whole KB at once — each concern has a small, natural scope.

Want Lens Trigger Scope it actually needs
Fill gaps (not in isolation) gap event question→gap gap + sibling gaps + open PRs
Don't rot — source changed source-sync event commit diff + docs citing that area
Don't rot — claims still true? verify patrol one file + its cited sources
Tidy — duplication / contradiction dedupe patrol one file + its k nearest neighbours
Tidy — too big / wrong responsibilities split patrol one file (+ neighbours)
Completeness ("what else?") complete patrol one file / one topic

The whole-KB crunch dissolves into patrol lenses, each on one file or one small neighbourhood. Even dedupe never goes O(n²): it compares the patrolled file only against its k nearest neighbours via the existing KB search.

03The architecture

Separate finding work from deciding to publish, with one shared gate in between.

Flow The top-level partition. Each flow has its own gap catalog, clusters, scheduled tasks and snapshot. Everything happens within a flow — the gate never reasons across flows.
File-set The concrete set of KB document paths a single PR writes to or deletes (writes[].path + deletes[]), inside one flow. A property of an individual PR, not something you configure. Two PRs overlap when their file-sets share any path. 
flow  (e.g. billing-kb)
 └── many docs
      └── a PR touches some subset of them   ← that subset is the "file-set"

Keying on the file-set is deliberate: keying on the flow is too coarse (one PR per flow serialises everything); keying on a single file is too fine (a split or dedupe spans files). The unit is "the set of files this change spans".

Rule 1

A signal is not a PR

Every lens emits a change intent { targets, lens, evidence, rationale } — never a branch directly.

Rule 2

One open PR per file-set

A PR is keyed by the files it touches. An intent overlapping an open PR folds into it — it can never spawn a competitor. File-path overlap is only a cheap pre-filter; an LLM does the actual fold/rewrite, so no mechanical text merge ever runs.

The current gap reconciler already implements Rule 1 (clusters → proposals). The redesign generalises that gate so source-sync and the patrols flow through it too.

04Two patrols, not one

Correctness and growth feel similar but behave oppositely — so they're separate jobs, run at different cadences and trusted differently.

Conservative

fix-patrol

Speaks only when something is demonstrably wrong. Silent on healthy files. PRs fix provable problems → quick to approve (still human-reviewed).

  • verify — a claim it can't prove
  • dedupe — a real duplicate / contradiction
  • split — a file that outgrew its responsibility
Proactive

improve-patrol

Grows docs that are fine-but-thin. Makes editorial calls → always wants human review. Runs on a slower cursor.

  • complete — "what else should we know here?"
The patrol trigger is a rolling cursor, not "all docs daily". Each tick: pick the N least-recently-checked files (or files past a staleness threshold). Rotates through the whole KB over days with bounded cost per tick.

05The reconcile gate

For each intent, the gate inspects existing open intents and open PRs, keyed by file/topic, then decides.

Situation Action
No overlap open a new PR
Overlaps an open, touchable PR (open & un-approved) fold into it — the default
Overlaps an open PR that's approved / merging defer to next round
Already covered / superseded drop

Overlap is detected cheaply, resolved by an LLM

hard file-set overlap — two PRs share a document path. Structural, exact, cheap.

soft topic overlap — two PRs touch different files about the same subject (refunds.md vs partial-refunds.md), caught by the dedupe lens's k-nearest-neighbour search. Heuristic.

Either signal only says two changes might collide. The LLM does the actual resolution — given the open PR's diff and the new intent's evidence, it rewrites the sections into one coherent change, attaches the intent as evidence, or rules them independent. Because a model is always in the loop, the soft signal can be generous and we never risk a bad mechanical merge.

Fold-on-overlap guard rail. Folding mutates an open PR — fine while it's un-approved. Once a reviewer has approved (or it's mid-merge), a new overlapping intent must not mutate it; that would silently invalidate the review. The rule is "fold if touchable, else defer", not "always fold".

06Why source-sync stays separate from verify

They look like the same "staleness" concern but pull in opposite directions — source-sync pushes (a commit landed → propagate to citing docs; event-driven, fast, small) while verify pulls (re-check whether claims are still provable, catching drift that didn't arrive via a tracked commit: renames, non-git sources, slow rot). Complementary, and they never collide — both emit intents into the same gate, so if source-sync already has a PR open on a file, verify just folds in.

07Migration sketch

Rough order that keeps the system working throughout.

  1. Introduce the intent type; make gap-reconciler emit/consume it explicitly (it already does in all but name).
  2. Generalise the reconcile gate to be lens-agnostic; add touchable/fold/defer logic.
  3. Route source-sync through the gate (emit intents instead of publishing branches).
  4. Build fix-patrol: rolling cursor + verify / dedupe / split lenses.
  5. Build improve-patrol (complete lens) on a slower cursor.
  6. Retire trigger_scheduled_crunch and the crunch_* job types once the patrols cover it.

Start at 1–2: the shared gate plus the intent type is the spine, and gap-reconciler.ts is already ~80% of it.

08Decisions

Folding & merge All patrol lenses fold into the existing open PR for the file-set. Nothing merges to main unattended — every PR is human-reviewed. fix vs improve PRs are labelled differently for easy triage; the merge gate is human for both.
Cursor fairness Oldest N + a random sample. Oldest-N clears the staleness backlog (exploit); the random share means nothing goes unvisited forever and load doesn't synchronise into waves (explore). Oldest-N dominates; random is the safety net.
Neighbourhood k Adaptive, not fixed — scope unknown today. Size k from the corpus (proportion of flow and/or a similarity threshold, not a hard count); tune with real numbers.

Still open: the exact oldest-N / random split (80/20?) and the concrete adaptivity rule for k — both settle once we have real volume.
